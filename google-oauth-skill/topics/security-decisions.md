# Security Decisions

## Spring Session vs JWT — Decision Record

JWT was considered and rejected in favor of Spring Session JDBC for browser-based apps. This is the reasoning, documented to avoid re-litigating it.

### What Spring Session gives you

- **httpOnly cookie** — JavaScript cannot read it (XSS protection)
- **SameSite=Lax** — CSRF protection built in
- **True server-side logout** — delete the session row in PostgreSQL/Redis and the token is immediately dead, before expiry, with no blocklist needed
- **Enterprise-grade** — battle-tested in production across thousands of Spring Boot apps
- **No client-side token management** — browser handles the cookie automatically

### Why JWT is redundant for browser-only apps

JWT's value proposition: "stateless signature verification is cheaper than a DB lookup."

This trade-off only makes sense when:
- You have **mobile apps or third-party callers** that can't use cookies
- You have **microservices** that need to pass identity across service boundaries without a shared session store
- You're building a **public API** for external developers

For a browser-only app calling its own backend over HTTPS, JWT adds complexity with no benefit:
- Token storage (localStorage = XSS risk; httpOnly cookie = you've just reinvented sessions)
- Expiry management and refresh token logic
- No true revocation without a blocklist (which requires a DB lookup — defeating the stateless benefit)

### Post-quantum cryptography makes JWT worse over time

JWT's stateless-crypto premise is broken by PQC:

| Algorithm | Signature size | Public key size |
|---|---|---|
| RS256 (current) | ~256 bytes | ~294 bytes |
| ES256 (current) | ~64 bytes | ~64 bytes |
| CRYSTALS-Dilithium (PQC) | 1–5 KB | 1–2 KB |
| FALCON (PQC) | 0.6–1.2 KB | 0.9–1.8 KB |

Multi-KB tokens in every HTTP request header, plus heavier CPU verification on every API call. At scale this is real cost — and it gets worse as PQC adoption grows.

Spring Session with Redis is unaffected by PQC. The session ID in the cookie stays a small random string forever. Cryptography lives at the TLS layer, not the application layer.

### The right answer if you need stateless tokens

Use Redis as the session store instead of PostgreSQL. Session lookup becomes a microsecond in-memory operation — faster than JWT signature verification even today, and unaffected by PQC in the future.

```yaml
spring:
  session:
    store-type: redis   # was: jdbc
```

Zero changes to controllers or services. Add `spring-session-data-redis` dependency. Done.

## Ownership Enforcement — The Missing Layer

Spring Session correctly blocks unauthenticated requests. What it does NOT do automatically is verify that the authenticated user owns the resource they're requesting.

Every service method that takes a user-scoped parameter should assert ownership:

```java
boolean owns = userAccountRepository.existsByUserIdAndCurrencyCode(
    currentUser.getUserId(), currencyCode);
if (!owns) throw new ResponseStatusException(HttpStatus.FORBIDDEN);
```

This closes the gap where a logged-in user could craft a request for another user's data if they know the right parameter values.

## API Keys (Future)

If programmatic access is needed (like Gemini, Stripe, GitHub), API keys coexist with session auth as a parallel Spring Security path:

```
Browser    → session cookie   → Spring Session validates → same endpoints
API client → X-Api-Key header → DB/cache lookup validates → same endpoints
```

Requirements when implementing:
- **Hash the key** — store only the hash in DB (never the raw key). User sees it once on generation.
- **Scopes** — read-only vs full access, per-resource restrictions
- **Rate limiting** — scripts can hit you thousands of times per second
- **Revocation** — users must be able to invalidate keys
- **Cache lookups** — validate via Redis cache, not a raw DB query on every call
- **Audit log** — which key called what and when
- **Ownership check** — same as above, even more critical when credentials are long-lived

## cert.pem on Production Machines

`cert.pem` (generated by `cloudflared tunnel login`) is the equivalent of your Cloudflare account credentials. It allows creating/deleting tunnels, rerouting DNS, and managing zones. It belongs only on developer machines — never on a server that only runs a tunnel.

For tunnel auth on production/shared machines, use `--token` only (see cloudflare-deployment-skill). The token is scoped to one tunnel, grants routing access only, and requires nothing in `~/.cloudflared`.
